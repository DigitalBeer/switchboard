# Sidebar Onboarding Assistant & Warm Startup Guard

## Goal
Implement a resilient onboarding assistant in the sidebar that activates when no Green-state agents are detected for 5 continuous seconds, surfacing a clear message and auto-expanding the "Terminal Operations" panel to guide the user into starting their agents. A session-scoped manual-collapse flag prevents the panel from re-opening after the user explicitly dismisses it.

## Metadata
**Tags:** frontend, UI
**Complexity:** Low

## User Review Required

> [!IMPORTANT]
> **Definition of Connected**: An agent is only considered "connected" if it is in a **Green** state (Ready and Registered). Orange (Recoverable) and Red (Offline) states do not count.
>
> **Grace Period**: The onboarding view activates only after a **5-second continuous disconnection**. This prevents the UI from "snapping" during brief terminal restarts or network stutters.
>
> **Session Memory**: If you manually collapse the "Terminal Operations" panel, the sidebar will **stay collapsed** for the remainder of your IDE session, even if agents are disconnected. This resets on IDE restart to ensure the feature remains useful for daily initialization.
>
> **Operational Scope**: This logic only runs when the initial plugin setup is complete (`needsSetup === false`). It does not interfere with the first-time bootstrapping process.

## Edge-Case & Dependency Audit

- **Race Conditions:** `renderAgentList()` is called from at least 11 callsites triggered by async events (`terminalStatuses`, `julesStatus`, re-render loops). The `setTimeout` follow-up scheduled during the grace period could stack if multiple rapid `terminalStatuses` events fire before the timeout executes. **Mitigation:** Use a single module-scope `_onboardingRetryTimer` handle; clear and reset it on every call that schedules the follow-up rather than accumulating multiple timers.
- **Security:** Pure client-side webview state; no IPC surface expansion. No risk.
- **Side Effects:** Auto-expanding the Terminal Operations panel triggers two `vscode.postMessage` calls (`getStartupCommands` and `getVisibleAgents`). These are already gated by the existing toggle listener and are idempotent. No additional side effects.
- **Dependencies & Conflicts:** The "Sidebar Panel Reorganization & Auto-Save" plan (`antigravity_380c6cb6...`) is in PLAN REVIEWED. If it restructures panel DOM IDs (`terminal-operations-fields`, `terminal-operations-chevron`, `terminal-operations-toggle`), those IDs must remain stable. **Document this dependency.** No active conflict since that plan is unimplemented. The "Append Design Doc" plan (`antigravity_bf878428...`) affects the planner prompt, not the sidebar shell — no conflict.

### Grumpy Critique

*"You. Yes, you. Let me explain the many ways this plan is an underdressed disaster.*

*First: **the timer is a lit fuse with no safety catch**. `renderAgentList()` is called from eleven — ELEVEN — separate event handlers, each of which can race. Every time `terminalStatuses` fires during the 5-second window, your plan schedules yet another `setTimeout`. By the time 5 seconds elapse, you'll have a busload of pending `renderAgentList()` invocations all firing simultaneously. Congratulations, you've built a rerender blizzard on the most latency-sensitive path in the UI. The fix takes THREE LINES — use a module-scope timer handle and `clearTimeout` it before scheduling — but the plan doesn't mention it. Inexcusable.*

*Second: **the plan calls `vscode.postMessage` for `getStartupCommands` and `getVisibleAgents` from inside `renderAgentList()`**. A render function should RENDER, not dispatch side-effecting messages. This is a latent footgun: if anything inside the extension responds to those messages by triggering a re-render (which `terminalStatuses` already does), you've built an accidental feedback loop. The correct fix is to fire those messages in the auto-expand block only when the panel is actually transitioning from closed to open — mirror the exact guard the toggle click handler already uses (`!fields.classList.contains('open')`).*

*Third: **the `isAgentGreen` helper is described but no code is shown**. The plan says it checks `lastDispatchReadiness[roleId]?.state === 'ready'`. But `createAgentRow` also has a fallback: terminal heartbeat liveness for agents without dispatch info. If you ignore that fallback in `isAgentGreen`, an agent that's alive via heartbeat but has no `dispatchReadiness` entry will be falsely counted as 'not green', and the user will see the onboarding message even when a healthy terminal is running. Wrong. Align with the actual `createAgentRow` logic.*

*Fix all three or ship nothing."*

### Balanced Response

**All three concerns are valid and addressed in the implementation below:**

1. **Timer accumulation → `_onboardingRetryTimer`**: A single module-scope variable holds the active timeout handle. Every path that would schedule a follow-up render first calls `clearTimeout(_onboardingRetryTimer)` before scheduling a new one. This collapses N concurrent timers into 1.

2. **Message dispatch from render → guarded transition**: The auto-expand block checks `!fields.classList.contains('open')` before posting messages — i.e., messages are only sent when the panel is genuinely transitioning from closed to open. This exactly mirrors the existing toggle click handler's behavior.

3. **`isAgentGreen` alignment**: The helper mirrors the full `createAgentRow` status chain: `dispatchState === 'ready'` OR `dispatchState === 'recoverable'` (Grumpy's concern — do we count orange? Per the spec: No. Orange is `recoverable`, not green. The helper explicitly excludes it.) OR terminal heartbeat liveness where no dispatch info is present (`dispatchState` is null and `termData.alive` is true and status is not `working/thinking`). Jules is always counted as green (its status dot is always green). **Correction from plan**: `recoverable` agents should NOT count as green per the User Review Required spec above. The helper only counts `state === 'ready'` and heartbeat-alive terminals.

**Change 3 — Add `isAgentGreen` helper** (insert after the `createAgentRow` function ends, before `renderAgentList` at line 3119):

```js
        /**
         * Returns true if roleId has an active Green-state agent.
         * Green = dispatchState 'ready', OR terminal heartbeat alive (any status including 'working').
         * Does NOT count 'recoverable' (orange) or 'not_ready' (red).
         * Jules is always green.
         * Analyst is always available (no terminal required).
         */
        function isAgentGreen(roleId) {
            if (roleId === 'jules') return true;
            if (roleId === 'analyst') return true; // always available
            const dispatchInfo = lastDispatchReadiness && roleId ? lastDispatchReadiness[roleId] : null;
            const dispatchState = dispatchInfo?.state ?? null;
            if (dispatchState === 'ready') return true;
            if (dispatchState === 'recoverable' || dispatchState === 'not_ready') return false;
            // Fallback: check terminal heartbeat for agents with no dispatch info
            const HEARTBEAT_THRESHOLD_MS = 60_000;
            const matchedTermName = Object.keys(lastTerminals).find(key => lastTerminals[key]?.role === roleId);
            if (!matchedTermName) return false;
            const termData = lastTerminals[matchedTermName];
            if (!termData) return false;
            const lastSeenMs = Date.parse(termData.lastSeen || '');
            const heartbeatAlive = !isNaN(lastSeenMs) && (Date.now() - lastSeenMs) < HEARTBEAT_THRESHOLD_MS;
            return termData.alive !== undefined ? !!termData.alive : (!!termData._isLocal || heartbeatAlive);
        }
```

- **Edge Cases Handled:**
  - **Timer accumulation**: `_onboardingRetryTimer` is cleared and reset on every grace-period evaluation. Only one timer is ever pending at a time.
  - **Panel IPC feedback loop**: The `!toFields.classList.contains('open')` guard ensures `getStartupCommands` / `getVisibleAgents` are only posted once per auto-expand event.
  - **User collapses then reconnects**: When an agent reconnects (`greenCount > 0`), `noAgentsDetectionTime` is reset. If the user later disconnects again, the 5-second grace period restarts. The `hasManuallyCollapsedThisSession` flag is *not* reset — so the panel won't auto-expand again for the rest of the session.
  - **Setup not complete**: `_setupComplete` gate prevents any of this logic from running during first-time onboarding.
  - **All agents hidden in Setup**: If all agents are hidden (`va.lead === false`, etc.) and `allRoles` is empty, `greenCount === 0` and the onboarding message would show. This is arguably correct since the sidebar would be empty anyway, but it's a degenerate case. **Clarification**: Add a guard — if `allRoles.length === 0`, treat as "nothing to show" and skip the onboarding guard (fall through to normal render which already shows "All agents hidden. Open Setup to configure.").
