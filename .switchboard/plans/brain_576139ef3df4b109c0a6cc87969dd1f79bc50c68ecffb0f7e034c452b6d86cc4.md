# Fixing Kanban Duplicates and Review Panel Boundary Escape

## Goal

Fix a bug where Kanban cards appear duplicated for plans originating from the Antigravity brain, and resolve the subsequent error when trying to review brain-based plans ("Plan file path is outside the workspace boundary"). Ensure brain plans produce a single Kanban card with a consistent `sessionId`, and that the review panel can open brain-sourced plan files without boundary rejection.

## Metadata

**Tags:** backend, database, bugfix
**Complexity:** High

## User Review Required

> **Security-sensitive changes**: The proposed modification to `_isPathWithinRoot` (L396–399 in TaskViewerProvider.ts) relaxes the workspace boundary check to whitelist the Antigravity brain directory (`~/.gemini/antigravity/brain`) and any configured custom plan folder. Additionally, ReviewProvider.ts (L398) performs a direct `writeFile` with **no path validation at all**, which is a path traversal vulnerability on the write path. Both changes require careful review to ensure no unintended file-system access is introduced.

## Root Cause Analysis

The duplicate kanban entries occur because two separate database records are created with **different `sessionId` values** for the same brain plan:

1. **`_registerPlan` (line ~4394)**: Sets `sessionId: entry.planId` (raw hash like `576139ef...`)
2. **`_mirrorBrainPlan` runsheet (line ~5622)**: Creates runsheet with `sessionId: antigravity_${pathHash}`

Since `sessionId` is the unique key in the database, these appear as two separate kanban cards. The sidebar dropdown disguises this by grouping duplicates by topic+date with counter suffixes (e.g., "Topic (2)").

The boundary error occurs when clicking the brain version because `_isPathWithinRoot` rejects absolute paths outside the workspace.

## Complexity Audit

### Routine

- **Updating `_isPathWithinRoot` to whitelist brain directory** (L396–399 in TaskViewerProvider.ts): Adding a known-safe directory to an existing check. The `_isPathWithin` helper (L4193–4197) already exists and uses `_getStablePath` for normalized comparison — this is a straightforward allowlist extension.

### Complex / Risky

- **Changing sessionId convention in `_registerPlan`** (L4384–4409 in TaskViewerProvider.ts): Currently stores `sessionId: entry.planId` (L4392). Changing brain plans to use `antigravity_${entry.planId}` affects all brain plan DB records. Existing duplicate records created under the old format will not be automatically reconciled — they become orphans in the database. The recovery flow at L4472 already uses `antigravity_${entry.planId}`, so this change aligns `_registerPlan` with the recovery path, but the transition requires a migration or cleanup step for pre-existing records.

- **UPSERT_PLAN_SQL conflict key mismatch** (KanbanDatabase.ts L113–131): The upsert conflicts on `plan_id` (PRIMARY KEY, L45), **not** on `session_id` (UNIQUE NOT NULL, L46). If an old DB record exists with `plan_id = "abc123"` and `session_id = "abc123"`, and the fix now tries to upsert `plan_id = "abc123"` with `session_id = "antigravity_abc123"`, the upsert will update the existing row (conflict on plan_id). However, if the runsheet path previously created a *separate* row with `plan_id = "antigravity_abc123"`, that row remains as an orphan. If a new insert attempts `session_id = "antigravity_abc123"` while the old runsheet row already holds that session_id with a different plan_id, a UNIQUE constraint violation on session_id will occur.

- **ReviewProvider path traversal vulnerability** (ReviewProvider.ts L398): `writeFile` is called with `this._currentPlan.planFileAbsolute` and has **no** boundary or path validation. This is currently marked "Optional" in the plan but represents a real path traversal risk — any plan record with a crafted absolute path could write to arbitrary locations on disk.

## Edge-Case & Dependency Audit

- **Race Conditions**: `_mirrorBrainPlan` (L5493–5650) and `_registerPlan` (L4384–4409) can be called in rapid succession by the file watcher. Both write to the database via `upsertPlans` (KanbanDatabase.ts L495–527). If `_registerPlan` fires before `_mirrorBrainPlan` completes, the sessionId must already be in its final `antigravity_`-prefixed form — otherwise a brief window exists where a non-prefixed record is written and then a second prefixed record follows, recreating the duplicate. The fix must set the sessionId consistently in `_registerPlan` *before* either path writes to the DB.

- **Security**: The `_isPathWithinRoot` relaxation must be strictly scoped to the exact brain directory (`~/.gemini/antigravity/brain`) via `_isPathWithin` (which uses normalized path comparison). It must not allow `..` traversal out of that directory. Symlink attacks are partially mitigated by `_getStablePath` normalization, but `fs.realpathSync` should be considered for the brain directory resolution to defeat symlinks pointing outside the allowlisted tree. ReviewProvider.ts L398 performs `writeFile` with zero path validation — this needs a boundary check equivalent to `_isPathWithinRoot` before any write operation.

- **Side Effects / Orphan Records**: Existing DB records with the old sessionId format (raw hash, no `antigravity_` prefix) will become orphans after the fix. These will not appear on the Kanban board if lookups switch to the new format, but they consume space and could cause confusion in diagnostics. A one-time migration query (`UPDATE plans SET session_id = 'antigravity_' || session_id WHERE session_id NOT LIKE 'antigravity_%' AND source_type = 'brain'`) or a startup cleanup routine is needed.

- **Dependencies & Conflicts**: No cross-plan conflicts identified. The "Sidebar Panel Reorganization" plan does not touch TaskViewerProvider's plan registration or boundary checking logic. The "Reverse plan dropdown order" plan modifies `getBoard()` in KanbanDatabase.ts but does NOT touch `upsertPlans` or sessionId logic.

## Adversarial Synthesis

### Grumpy Critique

1. **UPSERT conflict key mismatch is a ticking bomb.** The plan changes sessionId format but `UPSERT_PLAN_SQL` conflicts on `plan_id`, not `session_id`. If the old runsheet row has `plan_id = "antigravity_abc123"` and `session_id = "antigravity_abc123"`, and now `_registerPlan` tries to insert `plan_id = "abc123"` with `session_id = "antigravity_abc123"`, the plan_id won't conflict (different value) but the session_id UNIQUE constraint will blow up with a constraint violation. The plan doesn't address this at all.

2. **Orphan records are hand-waved away.** The plan says "upsertPlans updates rather than duplicates" but that's only true when plan_id matches. The old duplicate rows (created by the runsheet path with a different plan_id) won't be touched by the upsert — they'll sit in the database forever. No migration, no cleanup, no acknowledgment of the mess.

3. **ReviewProvider write vulnerability is "Optional"?!** A direct `writeFile` with an absolute path and zero validation is a textbook path traversal vulnerability. Marking it "Optional" in a security-sensitive bugfix plan is irresponsible. If an attacker can influence the `planFileAbsolute` field in the database, they can write arbitrary content to any path the process has access to.

4. **`path.relative` is not symlink-safe.** The existing `_isPathWithinRoot` uses `path.relative` which operates on string paths, not resolved filesystem paths. A symlink inside the brain directory pointing to `/etc/` would pass the allowlist check. The plan adds more allowlisted directories without addressing this fundamental weakness.

5. **No rollback story.** If the sessionId format change causes unexpected breakage in production, there's no documented way to revert the DB records back to their old format without data loss.

### Balanced Response

1. **UNIQUE constraint mitigation**: Before inserting with the new sessionId format, query for any existing row holding that session_id value (`getPlanBySessionId`). If found with a *different* plan_id, delete the orphan row first, then proceed with the upsert. This is already partially supported — `getPlanBySessionId` exists at KanbanDatabase.ts L664–673. Add a `deletePlanBySessionId` call before the upsert when a conflict is detected. Include this as an explicit step in the implementation.

2. **Orphan cleanup**: Add a startup migration routine in `KanbanDatabase.ts` that runs once after schema initialization: `DELETE FROM plans WHERE plan_id LIKE 'antigravity_%' AND session_id LIKE 'antigravity_%' AND EXISTS (SELECT 1 FROM plans p2 WHERE p2.session_id = plans.session_id AND p2.plan_id != plans.plan_id)`. Alternatively, the simpler approach: after the fix is deployed, the next `_registerPlan` call for each brain plan will correctly upsert by plan_id, updating the session_id on the canonical row. The runsheet-created orphan can be cleaned up by a one-time query at startup.

3. **ReviewProvider write validation**: Escalate from "Optional" to required. Add an equivalent `_isPathWithinRoot` check in ReviewProvider before the `writeFile` call at L398. This is a small change (3–5 lines) with high security impact. The plan should include this as a concrete code block, not a suggestion.

4. **Symlink hardening**: Use `fs.realpathSync` to resolve the brain directory and candidate path before comparison in `_isPathWithinRoot`. This converts symlinks to their real targets, preventing symlink-based escapes. The `_getStablePath` helper used by `_isPathWithin` already does some normalization but should be verified to call `realpathSync` for the allowlisted directories.

5. **Rollback**: Document a rollback SQL query in the Verification Plan: `UPDATE plans SET session_id = plan_id WHERE session_id LIKE 'antigravity_%'`. This reverts to the old format if needed. Include as a comment in the migration code.

## Proposed Changes

### TaskViewerProvider.ts

#### [MODIFY] `_registerPlan` (L4384–4409, sessionId assignment at L4392)

**Root fix**: Ensure brain plans use the same `sessionId` convention as their runsheet (`_mirrorBrainPlan` creates `runSheetId = \`antigravity_${pathHash}\`` at L5525). The recovery flow at L4472 already uses `antigravity_${entry.planId}`, so this aligns the primary registration path.

**Important**: Before upserting, check for an existing row holding the target session_id with a different plan_id (to avoid UNIQUE constraint violation on session_id, per KanbanDatabase.ts schema at L46). If found, delete the orphan row first.

```typescript
// Brain plans use 'antigravity_' prefix for sessionId to match their runsheet
const sessionId = entry.sourceType === 'brain' 
    ? `antigravity_${entry.planId}` 
    : entry.planId;
// Guard against UNIQUE constraint violation: if an orphan row holds this
// session_id with a different plan_id (from old runsheet path), remove it first
const existing = await db.getPlanBySessionId(sessionId);
if (existing && existing.planId !== entry.planId) {
    await db.deletePlanBySessionId(sessionId);
}
await db.upsertPlans([{
    planId: entry.planId,
    sessionId: sessionId,
    // ... rest of record
}]);
```

This ensures the brain plan registration and runsheet share the same `sessionId`, so `upsertPlans` updates rather than duplicates. The orphan guard prevents UNIQUE constraint violations when old-format records exist (see Adversarial Synthesis for details on the plan_id vs session_id conflict key mismatch).

#### [MODIFY] `_isPathWithinRoot` (L396–399)

**Secondary fix**: Allow brain directory paths and custom plan folder paths for plans that haven't been mirrored yet. Uses existing `_isPathWithin` helper (L4193–4197) which delegates to `_getStablePath` for normalized comparison.

```typescript
private _isPathWithinRoot(candidate: string, root: string): boolean {
    // Allow Antigravity brain directory (security: known safe location)
    const brainDir = path.join(os.homedir(), '.gemini', 'antigravity', 'brain');
    if (this._isPathWithin(brainDir, candidate)) return true;
    
    // Allow configured custom plan folder (if outside workspace)
    try {
        const config = vscode.workspace.getConfiguration('switchboard');
        const customFolder = config.get<string>('kanban.plansFolder')?.trim();
        if (customFolder) {
            const expanded = customFolder.startsWith('~')
                ? path.join(os.homedir(), customFolder.slice(1))
                : customFolder;
            const resolved = path.resolve(expanded);
            if (this._isPathWithin(resolved, candidate)) return true;
        }
    } catch { /* ignore config errors */ }
    
    const rel = path.relative(root, candidate);
    return !rel.startsWith('..') && !path.isAbsolute(rel);
}
```

### extension.ts (CRITICAL — Missing from original implementation)

#### [MODIFY] `isPathWithinRoot` (L305)

The `switchboard.reviewPlan` command uses a **separate** `isPathWithinRoot` function that must also be relaxed to allow brain directory paths. This is the source of the "Review plan path is outside the workspace boundary" error.

```typescript
function isPathWithinRoot(candidate: string, root: string): boolean {
    // Allow Antigravity brain directory
    const brainDir = path.join(os.homedir(), '.gemini', 'antigravity', 'brain');
    const relToBrain = path.relative(brainDir, candidate);
    if (!relToBrain.startsWith('..') && !path.isAbsolute(relToBrain)) return true;
    
    // Allow configured custom plan folder
    try {
        const config = vscode.workspace.getConfiguration('switchboard');
        const customFolder = config.get<string>('kanban.plansFolder')?.trim();
        if (customFolder) {
            const expanded = customFolder.startsWith('~')
                ? path.join(os.homedir(), customFolder.slice(1))
                : customFolder;
            const resolved = path.resolve(expanded);
            const relToCustom = path.relative(resolved, candidate);
            if (!relToCustom.startsWith('..') && !path.isAbsolute(relToCustom)) return true;
        }
    } catch { /* ignore */ }
    
    const rel = path.relative(root, candidate);
    return !rel.startsWith('..') && !path.isAbsolute(rel);
}
```

### ReviewProvider.ts (L398 — Required, not Optional)

**Path traversal fix**: The current code at L398 (`await fs.promises.writeFile(this._currentPlan.planFileAbsolute, content, 'utf8')`) performs a direct write with **no** path validation. Add a boundary check before the write operation, equivalent to `_isPathWithinRoot` in TaskViewerProvider.

```typescript
// Before writeFile at L398 — validate the target path
const planPath = this._currentPlan.planFileAbsolute;
if (!this._isPathWithinAllowedRoots(planPath)) {
    throw new Error(`Refusing to write: path "${planPath}" is outside allowed directories`);
}
await fs.promises.writeFile(planPath, content, 'utf8');
```

The `_isPathWithinAllowedRoots` helper should mirror the same allowlist logic as `_isPathWithinRoot` (workspace root, brain directory, custom plan folder).

## Review Feedback

### Stage 1 — Grumpy Principal Engineer Review

*Reviewer persona: The engineer who has seen every "simple fix" birth a P0 at 3am.*

---

**[NIT-1] Dual path-containment implementations with divergent normalization strategies**

So we have `_isPathWithinRoot` in TaskViewerProvider using the carefully crafted `_isPathWithin` helper (which calls `_getStablePath(path.resolve(...))` for normalized comparison with `startsWith`), and then ReviewProvider's `_isPathWithinAllowedRoots` rolls its own with raw `path.relative()`. Two security-critical path checks, two different algorithms, zero shared code.

Yes, `path.relative` does internal normalization. Yes, both approaches handle the `/a/b` vs `/a/bc` prefix-collision correctly. But when the next developer adds a fourth allowlisted root, they'll copy-paste from whichever file they happen to have open, and we'll have three different containment checks instead of two. Extract a shared utility or at minimum document why they differ.

**Practical risk**: Low — functionally equivalent on macOS/Linux. On Windows, `_getStablePath` lowercases but `path.relative` does not, meaning ReviewProvider's check could theoretically be bypassed with mixed-case paths. Since Switchboard currently targets macOS, this is academic. **Severity: NIT.**

---

**[NIT-2] Orphan guard + upsert is not a single atomic transaction**

`_registerPlan` (L4411-4415) performs three separate DB operations:
1. `getPlanBySessionId(sessionId)` — read
2. `deletePlan(sessionId)` — delete (auto-committed)
3. `upsertPlans([...])` — insert/update (own BEGIN/COMMIT)

If two `_registerPlan` calls interleave at the `await` boundaries (possible in the JS event loop), both could read the same orphan, both attempt delete (one succeeds, one no-ops), then both upsert. The second upsert would conflict on `plan_id` and update harmlessly, OR conflict on `session_id` if the plan_ids differ (UNIQUE violation → transaction rollback → logged error → self-heals on next attempt).

**Practical risk**: Extremely low — brain plan registration is infrequent and triggered by file watcher events that are deduplicated upstream. **Severity: NIT.**

---

**[NIT-3] Orphan metadata inheritance after delete — undocumented design decision**

After the orphan guard deletes a conflicting row, the code reuses the deleted row's `kanbanColumn`, `complexity`, `tags`, and `lastAction` as defaults for the new INSERT:

```typescript
kanbanColumn: existing?.kanbanColumn || 'CREATED',  // `existing` was just deleted!
```

This is *correct* for the brain-plan migration scenario (same logical plan, different DB key — preserves user's kanban progress). But it's non-obvious. A future maintainer seeing "use data from a deleted row" will rightfully suspect a bug. One comment explaining the intent would save an hour of archaeology. **Severity: NIT.**

---

**[NIT-4] No symlink resolution in either path check**

Neither `_isPathWithinRoot` nor `_isPathWithinAllowedRoots` calls `fs.realpathSync` before comparison. A symlink planted inside `~/.gemini/antigravity/brain/` pointing to `/etc/` would pass the containment check. The plan's own Adversarial Synthesis (§4) acknowledged this and recommended `realpathSync`. It wasn't implemented.

**Practical risk**: Low — requires the attacker to already have write access to the brain directory, at which point they have broader vectors. The plan explicitly deferred this. **Severity: NIT** (defer to hardening pass).

---

**[NIT-5] No startup orphan migration**

The plan's Balanced Response (§2) suggested a startup migration to clean up pre-existing orphan records created under the old `sessionId` format. This was not implemented. Instead, the orphan guard in `_registerPlan` handles the collision on next registration. Pre-existing orphans that are never re-registered will remain as invisible database rows forever.

**Practical risk**: Cosmetic — orphan rows consume negligible space and are invisible on the kanban board. `purgeOrphanedPlans` will eventually tombstone them if their files go missing. **Severity: NIT.**
