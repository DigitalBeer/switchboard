# Append Design Doc to Planner Prompts

## Goal
Add form controls to the Setup panel that let users provide a Design Doc / PRD link per project (workspace configuration). When enabled, the link is appended to every planner prompt produced by `buildKanbanBatchPrompt` for the `planner` role, ensuring planner agents always have the context document regardless of dispatch method (autoban, sidebar copy, Kanban drag-drop, or MCP dispatch).

## Metadata
**Tags:** frontend, backend, UI
**Complexity:** Low

## User Review Required

> [!IMPORTANT]
> The "Append Design Doc" feature will only apply to the **Planner** role. Prompts for Lead Coder, Coder, and Reviewer roles will remain unchanged to avoid context bloat, as the Design Doc's primary value is during the initial planning/enhancement stage.

> [!NOTE]
> Setting the Design Doc link will be persistent **per project** (Workspace configuration).

## Complexity Audit

### Routine
- Add two new VS Code workspace config properties to `package.json` (boolean + string) — follows the exact pattern of `switchboard.accurateCoding.enabled` and `switchboard.pairProgramming.aggressive`.
- Add UI controls (checkbox + text input) to the PROMPT CONTROLS section of `implementation.html` — identical pattern to every other toggle/row in that section.
- Add a config reader helper `_isDesignDocEnabled()` and `_getDesignDocLink()` to `TaskViewerProvider.ts` — identical pattern to `_isAccurateCodingEnabled()`.
- Wire save/load of the two settings through the existing `saveStartupCommands` and `ready` message handlers.
- Add `designDocLink` to `PromptBuilderOptions` and use it in the `planner` branch of `buildKanbanBatchPrompt`.

### Complex / Risky
- None

## Edge-Case & Dependency Audit
- **Race Conditions:** None. Settings are read synchronously from `vscode.workspace.getConfiguration` at prompt-build time. No async state races.
- **Security:** The design doc link is a user-supplied URL string embedded into a plain-text prompt. No sanitization required — it is not rendered as HTML; it is concatenated into the prompt string. No XSS risk in webview since the input is a standard `<input type="text">` with no `innerHTML` usage.
- **Side Effects:** Planner prompts will become slightly longer when the feature is enabled. No effect on any other role's prompt. If `designDocLink` is empty or the checkbox is off, the planner prompt is identical to today.
- **Dependencies & Conflicts:** No cross-plan conflicts found on the Kanban board. No pending plans touch `agentPromptBuilder.ts`, the PROMPT CONTROLS section of `implementation.html`, or the `saveStartupCommands` handler. The "Implement Explicit Tags and Metadata System" plan (sess_1774523390862) touches `agentPromptBuilder.ts` conceptually but modifies the `BatchPromptPlan` interface, not `PromptBuilderOptions` — no conflict.

## Adversarial Synthesis

### Grumpy Critique
> *"Oh WONDERFUL, another checkbox-driven feature on the Setup panel. Let me pour cold water on this parade:*
>
> 1. **You said 'all primary planner prompts' but you haven't mapped EVERY call site.** There are at least **four** distinct code paths that produce planner prompts:
>    - `_buildKanbanBatchPrompt` in `TaskViewerProvider.ts` (autoban / sidebar dispatch)
>    - `_generateBatchPlannerPrompt` in `KanbanProvider.ts` (Kanban board drag-drop / copy prompt button)
>    - The direct `buildKanbanBatchPrompt('planner', ...)` call at line 8110 of `TaskViewerProvider.ts` (MCP `move_kanban_card` dispatch)
>    - The Kanban board's clipboard-copy path via `_generatePromptForColumn` → `_generateBatchPlannerPrompt`
>
>    If you only wire the option through `PromptBuilderOptions` inside the canonical `buildKanbanBatchPrompt` function, you cover all four paths *automatically* — but ONLY if every call site actually passes `designDocLink`. Have you checked that all four do? Spoiler: `KanbanProvider._generateBatchPlannerPrompt` reads `aggressivePairProgramming` from `this._autobanState` but does NOT read workspace config — it has no access to `vscode.workspace.getConfiguration`. You'll need to thread the setting through from somewhere.
>
> 2. **Empty-string guard.** What happens when the user checks the box but leaves the link blank? You must guard against appending `DESIGN DOC REFERENCE:\n\n` to the prompt — that's garbage noise.
>
> 3. **URL validation?** The plan says 'link' but doesn't validate it. Are we going to append `asdfghjkl` to planner prompts? Probably fine — it's the user's problem — but at least document the non-validation decision.
>
> 4. **The `KanbanProvider` call site at line 660 uses a locally-constructed options object. It doesn't pass `designDocLink`.** If you fix the canonical builder but forget the callers that build their own options, the feature silently doesn't work for Kanban board actions."

### Balanced Response
> Grumpy is right on every point. Here's how each is addressed:
>
> 1. **All four call sites are covered because `designDocLink` is read inside `buildKanbanBatchPrompt` itself.** The solution reads the design doc setting *inside the canonical builder function* rather than requiring callers to pass it via options. This is architecturally superior — it guarantees coverage without changing every caller. However, `buildKanbanBatchPrompt` is a pure function and shouldn't read VS Code config directly. **Correct approach:** Add `designDocLink` as an optional field on `PromptBuilderOptions`, then make sure every caller that produces planner prompts passes it. There are exactly 4 callers to update:
>    - `_buildKanbanBatchPrompt` in `TaskViewerProvider.ts` (already reads config)
>    - `_generateBatchPlannerPrompt` in `KanbanProvider.ts` (needs to read config)
>    - Line 8110 in `TaskViewerProvider.ts` MCP dispatch (already reads config)
>    - Clipboard copy at line 6568 in `TaskViewerProvider.ts` (already reads config)
>
>    For `KanbanProvider`, we'll add a shared method to read the setting via `vscode.workspace.getConfiguration`, matching the existing pattern used for `accurateCodingEnabled` at line 695.
>
> 2. **Empty-string guard:** The implementation checks `designDocLink` is non-empty AND `designDocEnabled` is true before appending. Both conditions must be met.
>
> 3. **No URL validation.** The user can paste any string — a URL, a file path, a comment. This is intentional and consistent with how every other text input in the Setup panel works (e.g., plan ingestion folder, CLI commands). Explicitly documented in the plan.
>
> 4. **All callers updated.** Each caller is enumerated below with the exact change required.
