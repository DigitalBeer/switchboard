# Autonomous Quality Gate Harness

To solve the core issue of agents confirming work that either doesn't actually work or isn't built/deployed correctly, we need to inject a **"Ralph Loop"** (a tight feedback loop requiring verifiable output) into the agentic workflows. 

The strategy ensures agents are forced to test their own code and prove it works before you ever look at it, and ensures deployment commands are actually executed.

## Proposed Strategy

We can build this into Switchboard using three pillars:

### 1. Executable Verification Plans (The Benchmark)
We will modify the `/improve-plan` workflow (`.agent/workflows/improve-plan.md`). 
Currently, it creates a plan but often glosses over exactly *how* to test it. We will force the `improve-plan` workflow to append a rigid **`## Verification & Deployment Plan`** to every feature specification.
This will contain **exact terminal commands** (e.g., `npm run build`, test scripts, curl commands) and the expected output.

### 2. The Delegate Quality Gate (The Ralph Loop)
We will update the `switchboard_modes.md` (which enforces global rules for delegated agents) and `handoff-lead.md`.
We will add a non-negotiable directive for the executing agents:
* **The Proof Rule:** The agent is NOT allowed to state "done" or push the kanban card until they have actually copy-pasted the terminal output of the `Verification Plan` into their response.
* **The Deploy Rule:** If the system is a compiled frontend (like Vite/React) or requires Docker, the agent MUST run the build/deploy step as their final action. 

### 3. The Adversarial QA Workflow (Optional Add-on)
Instead of relying on the Coder to mark things done, we can create a dedicated `/qa` workflow (`.agent/workflows/qa.md`).
When a Coder finishes, they don't give the card to the user. They give it to the QA Agent. The QA Agent's sole purpose is to run the Verification commands. If it breaks, the QA agent automatically kicks it back to the Coder with the error log.
